### TASK-294: Preserve Closed Task Bodies in Quarterly Archive Shards
**Priority**: P1 (High)
**Estimate**: 3-5 hours

The current compact-ledger workflow removes full task definitions from the live
backlog when a task closes, which keeps active context small but also makes
closed task bodies hard to inspect unless they happened to be captured in an
older dated archive snapshot. Update task-closing workflow and archive-aware
lookup so every closed task preserves its full markdown block in a quarterly
archive shard such as `archive/closed_tasks/2026-Q1.md`, while keeping live
agent/default CLI reads strictly archive-free unless explicitly requested.

**Files**: `tasks/BACKLOG.md`, `tasks/COMPLETED.md`, `archive/closed_tasks/`, `src/horadus_cli/task_repo.py`, `src/horadus_cli/task_commands.py`, `src/core/docs_freshness.py`, `tests/`, `docs/AGENT_RUNBOOK.md`, `AGENTS.md`, `README.md`

**Acceptance Criteria**:
- [ ] Closing a task removes its full block from live `tasks/BACKLOG.md`, adds the compact completion entry to `tasks/COMPLETED.md`, and appends the full task block to the appropriate quarterly archive shard (for example `archive/closed_tasks/2026-Q1.md`)
- [ ] Archive shard format is stable and parseable enough for archive-aware CLI lookup without forcing agents to read archived task bodies by default
- [ ] `horadus tasks show`, `search`, and `context-pack` continue to stay live-only by default but can resolve closed-task bodies from `archive/closed_tasks/*.md` when archive access is explicitly requested
- [ ] Agent guidance/documentation state clearly that `archive/closed_tasks/` is non-authoritative for normal implementation flow and should only be consulted on explicit archive-aware requests
- [ ] Regression tests cover at least one close/archive path and one archive-aware lookup path using the new quarterly archive shard layout

### TASK-295: Enforce Pre-Merge Task Closure State
**Priority**: P1 (High)
**Estimate**: 3-5 hours

`TASK-292` exposed a workflow gap: a task PR can merge before the branch
contains the final backlog/current-sprint/completed cleanup, which then forces a
follow-up PR for bookkeeping that should have been part of the original task.
Add mechanical enforcement so `horadus tasks finish`, strict lifecycle
verification, and repo checks refuse to merge a task PR unless the primary
task's closure state is already present on the PR head.

**Files**: `src/horadus_cli/task_commands.py`, `src/horadus_cli/task_repo.py`, `tests/unit/test_cli.py`, `tests/`, `.github/workflows/`, `docs/AGENT_RUNBOOK.md`, `AGENTS.md`, `README.md`, `tasks/CURRENT_SPRINT.md`, `tasks/BACKLOG.md`, `tasks/COMPLETED.md`, `archive/closed_tasks/`

**Acceptance Criteria**:
- [ ] `horadus tasks finish TASK-XXX` blocks before merge when the primary task still appears open in live `tasks/BACKLOG.md` or `tasks/CURRENT_SPRINT.md`
- [ ] Strict lifecycle verification fails when the primary task is not yet recorded in `tasks/COMPLETED.md` and, after task-archive support lands, not yet preserved in the appropriate `archive/closed_tasks/*.md` shard
- [ ] The finish flow verifies that local HEAD, pushed branch HEAD, and PR head all match before merge so post-review local commits cannot be silently omitted from the merged task state
- [ ] The review gate distinguishes actionable current-head review feedback from stale, outdated, or already-resolved review threads so obsolete comments do not block merge completion
- [ ] A repo-side check (CI or equivalent) guards the same closure invariant so manual GitHub merges cannot bypass the requirement
- [ ] Regression tests cover both the blocking path (task still open, actionable current-head review feedback, or branch/PR head drift) and the pass path where ledger/archive closure is already on the PR head and stale/outdated review threads are ignored

**Estimate**: 1-2 days
**Exec Plan**: Required (`tasks/exec_plans/README.md`)

  others.

  work without changing the `horadus tasks ...` user interface.

### TASK-322: Harden `horadus tasks finish` Review Request Dedupe and Feedback Detection
**Priority**: P1 (High)
**Estimate**: 4-6 hours
**Spec**: `tasks/specs/322-harden-finish-review-dedupe-and-feedback-detection.md`
**Exec Plan**: Required (`tasks/exec_plans/TASK-322.md`)

Recent PR timelines show three finish-review workflow failures that undermine
the intended current-head review loop. First, the workflow can post a fresh
`@codex review` request on the PR's initial head even though the refresh path
should only request a new review after a later head change or same-head retry
condition. Second, same-head duplicate requests can still appear when a plain
`@codex review` comment already exists because Horadus only dedupes against its
own hidden-marker comments. Third, `horadus tasks finish` can miss relevant
new feedback because the review gate does not surface issue-comment review
results and the review-thread loader fails open when early `gh` metadata calls
fail.

Tighten the finish refresh contract so it requests review only for a qualifying
post-initial-head refresh, dedupes same-head requests across plain and
marker-tagged request comments, and fails closed when current-head feedback
state cannot be loaded. Preserve the current-head semantics from `TASK-307` /
`TASK-309`, but make the request/reporting behavior match the observed PR
history and the intended operator contract.

**Files**: `tools/horadus/python/horadus_workflow/pr_review_gate.py`, `tools/horadus/python/horadus_workflow/task_workflow_finish/_review_refresh.py`, `tools/horadus/python/horadus_workflow/task_workflow_finish/_review_threads.py`, `tools/horadus/python/horadus_workflow/task_workflow_finish/_review_window.py`, `tests/horadus_cli/v2/task_finish/`, `tests/unit/scripts/test_check_pr_review_gate.py`, `tasks/`

**Acceptance Criteria**:
- [ ] `horadus tasks finish` does not auto-request `@codex review` on the PR's initial head when there is no prior older-head review state or same-head timeout retry condition
- [ ] Same-head fresh-review dedupe treats an existing plain `@codex review` request and an existing marker-tagged Horadus request as equivalent blockers for another auto-request
- [ ] Current-head review reporting surfaces relevant Codex issue-comment results in addition to review objects/reactions so new comment-based feedback is not silently ignored
- [ ] Review-thread state loading fails closed when required PR/repo metadata cannot be read instead of silently treating the PR as thread-free
- [ ] Regression tests cover: initial-head no-request, same-head plain-comment dedupe, marker-comment dedupe, readable comment-based success/reporting, and fail-closed thread metadata errors
